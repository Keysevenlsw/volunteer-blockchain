# 数据库设计文档（新需求版）

## 1. 设计说明

本文档描述“基于区块链的志愿者服务认证系统”在新需求下的目标数据库设计。重构后的数据库遵循以下原则：

1. 志愿者默认归属组织与活动参与范围分离。
2. 活动发布审核与完成报告审核分表设计。
3. 积分账户全局唯一，积分流水按来源组织记录。
4. 商品统一进入全平台商城，商品审核与兑换履约分离。
5. 区块链采用“业务库索引 + 链上单条存证”的组合方案。

说明：

- 文档中的部分表为现有系统已存在表，部分为本次需求重构新增表。
- `users.role` 保留作为主角色兼容字段，实际后台权限通过 `roles / permissions / user_roles / role_permissions` 实现。
- `product_categories` 不纳入本次 v1 正式设计，如需扩展可后续增加。

## 2. RBAC 与用户域

### 2.1 公益组织表 `organizations`

```sql
CREATE TABLE organizations (
    organization_id INT AUTO_INCREMENT PRIMARY KEY,
    organization_name VARCHAR(255) NOT NULL,
    organization_description TEXT,
    avatar_path VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

字段说明：

- `organization_id`：组织主键
- `organization_name`：组织名称
- `organization_description`：组织简介
- `avatar_path`：组织头像路径
- `created_at`：创建时间

### 2.2 用户表 `users`

```sql
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(100) NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    email VARCHAR(100) NOT NULL UNIQUE,
    avatar_path VARCHAR(255),
    organization_id INT NULL,
    role ENUM('volunteer', 'organization_admin', 'system_admin', 'product_reviewer', 'activity_reviewer') NOT NULL,
    join_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    total_points INT NOT NULL DEFAULT 0,
    FOREIGN KEY (organization_id) REFERENCES organizations(organization_id)
);
```

字段说明：

- `organization_id`：默认归属组织，可为空，不限制活动参与范围
- `role`：主角色兼容字段，用于登录后默认身份识别
- `total_points`：用户全局积分余额

### 2.3 角色表 `roles`

```sql
CREATE TABLE roles (
    id INT AUTO_INCREMENT PRIMARY KEY,
    code VARCHAR(100) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    status TINYINT NOT NULL DEFAULT 1,
    is_system TINYINT(1) NOT NULL DEFAULT 0,
    data_scope VARCHAR(50) NOT NULL DEFAULT 'SELF',
    sort_order INT NOT NULL DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

系统固定角色建议初始化如下：

| code | name |
| --- | --- |
| `volunteer` | 志愿者 |
| `organization_admin` | 组织管理员 |
| `system_admin` | 超级管理员 |
| `product_reviewer` | 商品审核员 |
| `activity_reviewer` | 活动审核员 |

### 2.4 权限表 `permissions`

```sql
CREATE TABLE permissions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    code VARCHAR(100) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    module VARCHAR(100) NOT NULL,
    http_method VARCHAR(20) NOT NULL,
    api_pattern VARCHAR(255) NOT NULL,
    status TINYINT NOT NULL DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

说明：

- 用于定义后台接口级权限
- 例如活动审核、商品审核、用户角色分配、链上异常重试等能力

### 2.5 用户角色关联表 `user_roles`

```sql
CREATE TABLE user_roles (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    role_id INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uq_user_roles_user_role (user_id, role_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id),
    FOREIGN KEY (role_id) REFERENCES roles(id)
);
```

说明：

- 支持一个用户拥有多个后台角色

### 2.6 角色权限关联表 `role_permissions`

```sql
CREATE TABLE role_permissions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    role_id INT NOT NULL,
    permission_id INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uq_role_permissions_role_permission (role_id, permission_id),
    FOREIGN KEY (role_id) REFERENCES roles(id),
    FOREIGN KEY (permission_id) REFERENCES permissions(id)
);
```

### 2.7 默认归属组织申请表 `organization_join_requests`

```sql
CREATE TABLE organization_join_requests (
    id INT AUTO_INCREMENT PRIMARY KEY,

```

说明：

- 用于记录志愿者申请加入默认归属组织的过程
- 审核通过后写入 `users.organization_id`

## 3. 活动发布、参与与审核域

### 3.1 活动表 `activities`

```sql
CREATE TABLE activities (
    activity_id INT AUTO_INCREMENT PRIMARY KEY,
    organization_id INT NOT NULL,
    activity_name VARCHAR(255) NOT NULL,
    description TEXT,
    start_date DATETIME NOT NULL,
    end_date DATETIME NOT NULL,
    publish_date DATETIME NULL,
    location VARCHAR(255) NOT NULL,
    contact_name VARCHAR(255),
    contact_phone VARCHAR(20),
    category_tags VARCHAR(255),
    image_path VARCHAR(255),
    max_participants INT NOT NULL DEFAULT 1,
    current_participants INT NOT NULL DEFAULT 0,
    enroll_deadline DATETIME NULL,
    created_by INT NOT NULL,
    requested_reward_points INT NOT NULL DEFAULT 0,
    approved_reward_points INT NULL,
    review_status ENUM('draft', 'pending_review', 'approved', 'rejected') NOT NULL DEFAULT 'draft',
    review_note VARCHAR(500),
    reviewed_by INT NULL,
    reviewed_at DATETIME NULL,
    status ENUM('pending', 'ongoing', 'completed') NOT NULL DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (organization_id) REFERENCES organizations(organization_id),
    FOREIGN KEY (created_by) REFERENCES users(user_id),
    FOREIGN KEY (reviewed_by) REFERENCES users(user_id)
);
```

字段说明：

- `requested_reward_points`：组织申请积分
- `approved_reward_points`：平台最终批准积分
- `review_status`：活动发布审核状态，不与业务状态混用
- `status`：活动业务状态，仅表示待开始、进行中、已完成

业务约束：

1. 只有 `review_status = 'approved'` 的活动才能发布给志愿者。
2. 修改关键字段后，系统应自动将 `review_status` 重置为 `pending_review`。
3. 积分发放只读取 `approved_reward_points`。

### 3.2 活动发布审核记录表 `activity_publish_reviews`

```sql
CREATE TABLE activity_publish_reviews (
    id INT AUTO_INCREMENT PRIMARY KEY,
    activity_id INT NOT NULL,
    reviewer_id INT NOT NULL,
    reviewer_role ENUM('activity_reviewer', 'system_admin') NOT NULL,
    review_action ENUM('submitted', 'approved', 'rejected', 'escalated') NOT NULL,
    requested_reward_points INT NOT NULL DEFAULT 0,
    approved_reward_points INT NULL,
    review_note VARCHAR(500),
    is_escalated TINYINT(1) NOT NULL DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (activity_id) REFERENCES activities(activity_id),
    FOREIGN KEY (reviewer_id) REFERENCES users(user_id)
);
```

说明：

- 记录活动提审、初审、升级复审、通过、驳回全过程
- `is_escalated = 1` 表示活动被提交至超级管理员复审

### 3.3 活动参与表 `activity_participation`

```sql
CREATE TABLE activity_participation (
    participation_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    activity_id INT NOT NULL,
    earned_points INT NOT NULL DEFAULT 0,
    participation_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uq_activity_participation_user_activity (user_id, activity_id),
    FOREIGN KEY (user_id) REFERENCES users(user_id),
    FOREIGN KEY (activity_id) REFERENCES activities(activity_id)
);
```

说明：

- 志愿者可参与任意组织已审核活动
- `earned_points` 为该次活动最终发放积分

### 3.4 活动完成报告表 `activity_completion`

```sql
CREATE TABLE activity_completion (
    completion_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    activity_id INT NOT NULL,
    completion_status ENUM('pending', 'approved', 'rejected') NOT NULL DEFAULT 'pending',
    report_text TEXT,
    service_location VARCHAR(255) NOT NULL,
    service_start_time DATETIME NOT NULL,

    contribution_details TEXT,
    attachment_paths TEXT,

    approved_by INT NULL,

    FOREIGN KEY (user_id) REFERENCES users(user_id),
    FOREIGN KEY (activity_id) REFERENCES activities(activity_id),
    FOREIGN KEY (approved_by) REFERENCES users(user_id)
);
```

说明：

- 由志愿者提交活动完成报告
- 由活动所属组织管理员审核

### 3.5 完成报告审核记录表 `completion_reviews`

```sql
CREATE TABLE completion_reviews (
    id INT AUTO_INCREMENT PRIMARY KEY,
    completion_id INT NOT NULL,

```

说明：

- 记录组织管理员对完成报告的审核历史

## 4. 积分治理与积分流水域

### 4.1 积分流水表 `points_records`

```sql
CREATE TABLE points_records (
    record_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    organization_id INT NOT NULL,
    points INT NOT NULL,
    transaction_type ENUM('earned', 'spent') NOT NULL,
    source VARCHAR(255) NOT NULL,
    reference_type VARCHAR(50),
    reference_id INT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(user_id),
    FOREIGN KEY (organization_id) REFERENCES organizations(organization_id)
);
```

说明：

- `organization_id` 表示积分来源组织或商品归属组织
- 该字段不代表独立积分账户
- 积分账户只有一个，统一归属到用户

### 4.2 积分规则配置表 `points_rule_configs`

```sql
CREATE TABLE points_rule_configs (
    id INT AUTO_INCREMENT PRIMARY KEY,
    activity_category VARCHAR(100) NOT NULL,
    min_service_hours DECIMAL(6,2) NOT NULL,
    max_service_hours DECIMAL(6,2) NOT NULL,
    suggested_points INT NOT NULL,
    max_points INT NOT NULL,
    escalation_threshold INT NOT NULL,
    status TINYINT NOT NULL DEFAULT 1,
    created_by INT NOT NULL,
    updated_by INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (created_by) REFERENCES users(user_id),
    FOREIGN KEY (updated_by) REFERENCES users(user_id)
);
```

说明：

- 用于配置不同活动类型和服务时长下的建议积分、最高积分与复审阈值
- 供活动审核员与超级管理员判断活动申请积分是否合理

积分发放规则：

1. 完成报告审核通过后，系统按 `activities.approved_reward_points` 发放积分。
2. 若活动未通过发布审核，则活动不能进入报名流程，也不能产生积分。
3. 若兑换取消，系统需回滚积分并记录返还流水。

## 8. 关键约束与规则

### 8.1 审核隔离约束

1. 活动审核人不能审核自己所属组织提交的活动。
2. 商品审核人不能审核自己所属组织提交的商品。
3. 组织管理员只能审核本组织活动对应的完成报告。

### 8.2 活动审核约束

1. 组织提交活动时只能填写 `requested_reward_points`。
2. 最终对外发布和积分发放使用 `approved_reward_points`。
3. 活动关键字段变更后 `review_status` 自动置为 `pending_review`。

### 8.3 商品审核约束

1. 商品提交后进入 `pending_review`。
2. 商品审核通过后才允许进入统一商城。
3. 商品下架需记录在 `product_reviews` 中。

### 8.4 积分约束

1. 用户只有一个全局积分账户。
2. 积分来源可来自不同组织活动，但统一累计到 `users.total_points`。
3. 兑换取消时必须执行积分返还和库存恢复。

### 8.5 存证约束

1. 仅完成报告审核通过后才允许创建链上存证索引。
2. 链上索引必须记录业务类型、业务编号、摘要和上链状态。
3. 上链失败记录必须保留，便于后台重试。

## 9. 与接口设计的对应关系

数据库设计需要支持以下接口能力：

1. `GET /api/activities/volunteer`
   - 查询 `activities` 中 `review_status = 'approved'` 的活动
2. `POST /api/participations/{activityId}`
   - 写入 `activity_participation`
3. `POST /api/completions`
   - 写入 `activity_completion`
4. `GET /api/admin/activity-reviews`
   - 查询 `activities.review_status = 'pending_review'`
5. `POST /api/admin/activity-reviews/{activityId}`
   - 更新 `activities` 并写入 `activity_publish_reviews`
6. `GET /api/admin/product-reviews`
   - 查询 `products.review_status = 'pending_review'`
7. `POST /api/admin/product-reviews/{productId}`
   - 更新 `products` 并写入 `product_reviews`
8. `GET /api/public/evidences`
   - 联查 `blockchain_evidences` 与业务表输出公开存证列表

## 10. 总结

本次数据库重构以“跨组织参与、统一商城、平台审核、积分治理、公开存证”为主线，将原有单组织活动和独立商城模型升级为平台化治理模型。通过增加活动发布审核表、商品审核表、积分规则配置表，并完善活动表和商品表中的审核字段，系统能够更有效地控制高积分风险、提升业务透明度，并为后续接口扩展和系统实现提供稳定的数据结构基础。

## 11. 链上积分系统补充设计

### 11.1 `points_records` 链上索引字段

积分余额和积分交易以 `VolunteerPointsLedger` 合约为准，`points_records` 作为链上积分交易索引表，需要补充以下字段：

```sql
ALTER TABLE points_records
    MODIFY COLUMN organization_id INT NULL,
    ADD COLUMN biz_key VARCHAR(255),
    ADD COLUMN digest VARCHAR(128),
    ADD COLUMN tx_hash VARCHAR(255),
    ADD COLUMN block_number VARCHAR(64),
    ADD COLUMN contract_name VARCHAR(255),
    ADD COLUMN onchain_status ENUM('pending', 'success', 'failed') NOT NULL DEFAULT 'success',
    ADD COLUMN error_message VARCHAR(1000),
    ADD COLUMN onchain_at DATETIME NULL,
    ADD COLUMN chain_balance_after INT;
```

新增索引：

```sql
CREATE UNIQUE INDEX uq_points_records_biz_key
ON points_records(biz_key);

CREATE INDEX idx_points_records_onchain_status
ON points_records(onchain_status, created_at);
```

### 11.2 用户积分余额缓存

`users.total_points` 不再作为积分系统的最终权威，只保存链上余额缓存。完成报告发放积分、商城扣减积分、取消兑换退款均以链上合约交易成功为前提，链成功后再回写该缓存字段。

### 11.3 链上业务键

积分交易业务键必须全局唯一，用于合约幂等校验和数据库唯一索引：

1. 活动发放积分：`completion:{completionId}:earn:{userId}`。
2. 商城兑换扣分：`redemption:{redemptionId}:spend:{userId}`。
3. 兑换取消退款：`redemption:{redemptionId}:refund:{userId}`。
4. 历史积分迁移：`migration:init:{userId}`。

### 11.4 新增接口对应关系

1. `GET /api/points/balance`
   - 查询链上余额并与 `users.total_points` 缓存对比。
2. `POST /api/admin/points/sync-balances`
   - 超级管理员触发链上余额同步，修正 `users.total_points`。
3. `POST /api/admin/points/migrate`
