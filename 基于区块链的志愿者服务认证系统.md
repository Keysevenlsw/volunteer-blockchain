# 基于区块链的志愿者服务认证系统设计方案（新需求版）

## 1. 项目概述

### 1.1 建设背景

传统志愿服务认证普遍存在活动真实性难校验、服务记录易被篡改、积分激励规则不透明、组织之间标准不统一等问题。为解决上述痛点，系统采用区块链存证与平台审核结合的方式，构建一个可审计、可追溯、可激励的志愿服务认证平台。

### 1.2 建设目标

本系统以“基于区块链的志愿者服务认证系统”为目标，围绕志愿活动发布、参与、审核、积分发放、商品兑换和链上存证，形成完整的业务闭环。系统重构后的核心目标如下：

1. 志愿者可参与所有公益组织发布并通过审核的志愿活动，不再受默认归属组织限制。
2. 平台仅保留一个统一积分商城，志愿者使用全局积分账户进行兑换。
3. 所有通过审核的完成报告均可生成链上存证，支持公开查询与验证。
4. 通过平台审核控制活动积分风险，避免公益组织恶意设置高积分活动谋取私利。
5. 通过 RBAC 多角色权限模型实现“组织提交、平台审核、链上存证、统一监管”的治理体系。

### 1.3 技术路线

系统采用以下技术栈：

- 后端：`Spring Boot`
- 前端：`Vue + Element Plus`
- 区块链平台：`FISCO BCOS`
- 链上交互与运维：`WeBASE`
- 数据库：`MySQL`

系统明确采用 `FISCO BCOS + WeBASE` 实现链上存证，不再使用“Fabric 或以太坊二选一”的宽泛描述。

## 2. 总体业务规则

重构后的系统遵循以下统一业务规则：

1. 志愿者仍可申请加入一个默认归属组织，但默认归属组织只用于组织身份归属与个人展示，不限制志愿者参与其他组织活动。
2. 公益组织发布的活动必须先经过平台活动审核流程，审核通过后才能发布给志愿者报名。
3. 公益组织提交的商品必须经过平台商品审核流程，审核通过后才能进入全平台统一积分商城。
4. 志愿者积分账户为全局唯一账户，积分可来自不同组织活动，但统一累计在用户维度。
5. 活动积分采用“申请积分 + 批准积分”双字段机制，最终积分发放只能依据批准积分。
6. 志愿者完成活动后，由活动所属组织审核完成报告；审核通过后再进行链上存证与积分发放。
7. 全平台存证信息向公众开放查询，但仅展示必要公开信息，不泄露用户隐私数据。

## 3. RBAC 角色设计

系统固定采用五类角色，并允许后台通过 RBAC 为用户分配多个角色。

| 角色编码 | 中文名称 | 职责说明 |
| --- | --- | --- |
| `volunteer` | 志愿者 | 浏览全平台已审核活动、报名活动、提交完成报告、查看积分与兑换记录、查看公开存证 |
| `organization_admin` | 组织管理员 | 维护本组织资料、创建并提交活动、提交商品、审核本组织活动完成报告、处理本组织商品兑换 |
| `activity_reviewer` | 活动审核员 | 审核公益组织提交的活动，核验活动真实性、服务时长、人数配置与申请积分合理性 |
| `product_reviewer` | 商品审核员 | 审核公益组织提交的商品，决定其是否进入全平台统一积分商城 |
| `system_admin` | 超级管理员 | 管理用户与角色、维护积分规则、复审高风险活动、处理异常组织、处理系统级链上异常 |

### 3.1 角色设计原则

1. 平台审核角色与组织业务角色分离，避免组织自行决定活动上线和商品上架。
2. 同一用户可以拥有多个后台角色，但必须满足职责隔离要求。
3. 活动审核员与超级管理员不得审核自己所属组织提交的活动。
4. 商品审核员不得审核自己所属组织提交的商品。
5. `system_admin` 拥有系统最高管理权限，并作为高风险活动复审角色和异常处理兜底角色。

## 4. 功能需求分析

### 4.1 志愿者需求

1. 可浏览所有公益组织发布且通过审核的志愿活动。
2. 可跨组织报名活动，不要求先加入活动所属组织。
3. 可查看个人参与记录、完成报告、积分流水与兑换记录。
4. 可在统一积分商城中使用全局积分余额兑换商品。
5. 可查看链上的所有志愿活动存证及其校验状态。
6. 可申请加入一个默认归属组织，用于组织身份归属展示与组织互动。

### 4.2 组织管理员需求

1. 可维护本组织基础信息。
2. 可创建活动草稿并提交平台审核。
3. 可为活动填写申请积分，但不能直接决定最终发放积分。
4. 可查看本组织活动的发布审核结果和驳回意见。
5. 可审核本组织志愿者提交的活动完成报告。
6. 可提交商品进入统一积分商城，管理本组织商品库存。
7. 可处理本组织商品的兑换履约与取消。

### 4.3 活动审核员需求

1. 审核公益组织提交的活动。
2. 核查活动时间、地点、人数、活动类型和申请积分是否合理。
3. 对正常风险活动可直接审核通过或驳回。
4. 对高积分活动、命中异常规则的活动提交超级管理员复审。
5. 记录审核意见、建议积分与风险标记。

### 4.4 商品审核员需求

1. 审核公益组织提交的商品是否可进入统一积分商城。
2. 对违规、虚假或不适合的平台商品进行驳回。
3. 记录商品审核意见与审核时间。

### 4.5 超级管理员需求

1. 管理用户、角色和权限。
2. 配置积分规则与复审阈值。
3. 复审高风险活动和异常积分申请。
4. 管理异常组织、冻结违规活动或下架违规商品。
5. 查看全平台链上存证与失败重试记录。

## 5. 核心业务流程

### 5.1 默认归属组织申请流程

1. 志愿者注册后可选择申请加入一个默认归属组织。
2. 组织管理员审核加入申请。
3. 审核通过后，用户资料中写入默认归属组织。
4. 默认归属组织不影响志愿者参加其他组织活动。

### 5.2 活动发布审核流程

1. 组织管理员创建活动草稿，填写活动基础信息和 `申请积分`。
2. 活动进入 `draft` 状态，可继续编辑。
3. 组织提交活动后，活动进入 `pending_review` 状态。
4. 活动审核员进行初审，审核内容包括：
   - 活动是否真实、完整、明确
   - 服务时长与人数配置是否合理
   - 活动类型是否匹配申请积分
   - 是否命中异常风控规则
5. 若活动正常且申请积分在合理范围内，活动审核员可直接审核通过，并写入 `批准积分`。
6. 若活动申请积分超过系统建议值，或命中高风险规则，则活动审核员提交超级管理员复审。
7. 超级管理员复审通过后，活动才允许正式发布。
8. 审核通过后，活动进入 `approved` 状态，并对志愿者开放展示与报名。
9. 若活动被驳回，记录驳回原因，组织管理员可修改后重新提交。

### 5.3 志愿者参与与完成报告流程

1. 志愿者可浏览全平台所有已审核发布活动。
2. 志愿者选择任意活动报名，系统校验名额、时间和重复报名情况。
3. 志愿者完成活动后，提交完成报告，内容包括服务地点、服务起止时间、工作总结、贡献内容和附件。
4. 活动所属组织管理员审核完成报告。
5. 审核通过后，系统进入积分发放与链上存证流程。
6. 审核不通过时，记录驳回原因并允许志愿者重新提交。

### 5.4 积分发放流程

1. 完成报告审核通过后，系统读取活动的 `批准积分`。
2. 积分仅按 `批准积分` 发放，不得直接使用 `申请积分`。
3. 积分发放后写入用户全局积分账户。
4. 同时生成积分流水，记录积分来源组织、业务类型和关联业务编号。

### 5.5 统一商城流程

1. 组织管理员创建商品并提交平台审核。
2. 商品审核员审核商品，审核通过后进入统一积分商城。
3. 志愿者浏览统一积分商城，使用全局积分余额兑换商品。
4. 商品兑换后由商品所属组织处理履约。
5. 如组织取消兑换，系统回滚积分并恢复库存。

### 5.6 公开存证查询流程

1. 完成报告审核通过后，系统提取业务摘要并生成 `digest`。
2. 平台先将摘要写入业务库中的链上索引表。
3. 系统通过 `WeBASE` 调用 `FISCO BCOS` 合约写链。
4. 写链成功后，记录交易哈希、区块高度和上链时间。
5. 公众可查看全平台存证列表和详情。
6. 当上链失败时，后台角色可进行异常重试。

## 6. 积分治理与风控策略

活动积分是平台治理重点，因此系统采用“规则配置 + 平台审核 + 风险复审”的管控机制。

### 6.1 基本规则

1. 组织只能填写 `申请积分`，不能直接控制最终积分发放。
2. 系统维护统一的积分规则表，按活动类型、服务时长和人数规模给出建议积分与最高积分。
3. 平台活动审核员必须核对申请积分与规则建议值之间的差异。
4. 当活动申请积分超过建议值或命中高风险条件时，必须提交超级管理员复审。
5. 活动修改关键字段后必须重新进入待审核状态。

### 6.2 关键字段重新审核规则

以下字段一旦在活动审核通过后被修改，系统必须自动将活动重新置为 `pending_review`：

- 活动开始时间
- 活动结束时间
- 活动地点
- 最大参与人数
- 活动类型或标签
- 申请积分

### 6.3 风险判定示例

系统至少应支持以下异常判定：

1. 短时长高积分活动。
2. 同一组织短期内频繁提交高积分活动。
3. 重复内容但积分明显偏高的活动。
4. 同一批志愿者集中参与某组织高积分活动。
5. 人数极少但积分异常偏高的活动。

### 6.4 审核隔离要求

1. 活动审核员与超级管理员不得审核自己所属组织提交的活动。
2. 商品审核员不得审核自己所属组织提交的商品。
3. 组织管理员仅能审核本组织志愿者提交的完成报告，不能审核平台发布审核流程。

## 7. 系统实现方案

### 7.1 总体架构

系统采用“前端层 + 后端层 + 数据层 + 区块链层”四层结构：

1. 前端层：基于 `Vue + Element Plus` 实现志愿者端、组织端、管理员端和公开查询页面。
2. 后端层：基于 `Spring Boot` 实现认证、RBAC、活动管理、审核流程、积分流水、商城兑换和公开查询接口。
3. 数据层：基于 `MySQL` 存储业务数据、审核数据、积分数据和链上索引数据。
4. 区块链层：基于 `FISCO BCOS + WeBASE` 完成链上存证写入、交易哈希记录和异常重试。

### 7.2 前端实现方案

#### 志愿者端

1. 活动大厅改为全平台活动列表，支持按组织、状态、关键词筛选。
2. 个人中心展示默认归属组织、参与记录、完成报告、积分流水和兑换记录。
3. 商城页面改为统一积分商城，仅展示审核通过商品。
4. 新增公开存证页，支持列表浏览与详情查询。

#### 组织管理员端

1. 新增活动草稿、提审、审核结果查看等功能。
2. 新增商品提审、审核结果查看等功能。
3. 保留完成报告审核与商品兑换处理能力。

#### 平台管理端

1. 新增活动审核工作台。
2. 新增商品审核工作台。
3. 新增积分规则维护界面。
4. 新增链上异常监控与重试界面。

### 7.3 后端实现方案

1. 活动服务拆分为“组织提交”和“平台审核”两个阶段。
2. 商品服务拆分为“组织维护”和“平台审核”两个阶段。
3. 完成报告审核仍由活动所属组织管理员处理。
4. 公开服务新增存证列表接口与存证详情接口。
5. RBAC 支持多角色分配，并按接口级权限控制后台能力。

### 7.4 区块链实现方案

1. 保持当前 `VolunteerEvidenceRegistry` 单条存证合约模式不变。
2. 合约负责单条业务数据的写链与按业务键查询。
3. 全平台公开存证列表不依赖链上遍历，而依赖业务库中的链上索引表。
4. 后端对完成报告、活动和志愿者信息生成 `SHA-256` 摘要，并通过 WeBASE 写入 FISCO BCOS。

## 8. 接口与类型设计说明

以下为本次需求重构后需要体现在后端设计中的主要对外能力：

### 8.1 活动与参与接口

- `GET /api/activities/volunteer`
  - 语义改为获取全平台已审核活动列表。
- `POST /api/participations/{activityId}`
  - 取消“必须属于同一组织”的限制。
- `POST /api/completions`
  - 志愿者提交完成报告，由活动所属组织审核。

### 8.2 活动审核接口

- `GET /api/admin/activity-reviews`
  - 查询待审核活动。
- `POST /api/admin/activity-reviews/{activityId}`
  - 审核活动，支持通过、驳回、升级复审。

### 8.3 商品审核接口

- `GET /api/admin/product-reviews`
  - 查询待审核商品。
- `POST /api/admin/product-reviews/{productId}`
  - 审核商品，支持通过、驳回、下架。

### 8.4 公开存证接口

- `GET /api/public/evidences`
  - 公开查询全平台存证列表。
- `GET /api/public/evidences/{bizType}/{bizId}`
  - 查看指定业务的存证详情。

### 8.5 主要 VO/DTO 变化

1. `ActivityVO` 需要新增或映射以下字段：
   - `requestedRewardPoints`
   - `approvedRewardPoints`
   - `reviewStatus`
2. `ProductVO` 需要新增：
   - `reviewStatus`
3. 新增活动审核请求对象：
   - `status`
   - `reviewNote`
   - `approvedRewardPoints`
4. 新增商品审核请求对象：
   - `status`
   - `reviewNote`

## 9. 非功能需求

### 9.1 安全性

1. 所有用户接口均需进行身份认证和权限校验。
2. 平台审核接口需记录审核日志，确保可审计。
3. 对公开存证接口仅开放必要字段，不公开邮箱、密码等敏感信息。
4. 上传附件应进行类型、大小和路径控制。

### 9.2 可追溯性

1. 活动发布审核过程需留痕。
2. 商品审核过程需留痕。
3. 完成报告审核过程需留痕。
4. 积分发放、积分消费、积分回滚均需可追溯。
5. 链上存证需支持摘要校验与交易哈希查询。

### 9.3 可扩展性

1. 积分规则通过配置表维护，便于后续调整。
2. RBAC 支持新增角色和权限。
3. 公开存证接口可扩展分页、筛选和导出能力。

## 10. 测试与验收要点

1. 志愿者未加入任何组织时，仍可浏览并报名已审核活动。
2. 活动在平台审核通过前，不得出现在志愿者活动大厅。
3. 高积分活动必须进入超级管理员复审流程。
4. 活动关键字段变更后必须重新审核。
5. 完成报告审核通过后，积分发放值必须等于 `批准积分`。
6. 商品审核通过前不得进入统一商城。
7. 公开存证列表和详情可正常查看摘要、交易哈希、区块高度和校验结果。

## 11. 总结

本次需求重构后，系统从“组织内活动与组织独立商城”的模式升级为“跨组织参与、统一积分商城、平台审核治理、公开链上存证”的平台化模式。通过引入五类角色、活动审核机制、商品审核机制、积分治理规则和公开存证能力，系统能够在保证透明可信的同时，显著降低公益组织恶意设定高积分活动带来的业务风险，为志愿服务认证提供更规范、更可持续的技术方案。

## 12. 链上积分系统补充设计

### 12.1 积分权威口径

新版本中，积分余额和积分交易以 FISCO BCOS 链上 `VolunteerPointsLedger` 合约为最终权威。MySQL 中的 `users.total_points` 只作为链上余额缓存，`points_records` 只作为链上积分交易的业务索引和查询加速表。

### 12.2 链上积分交易

链上积分合约固定支持以下能力：

1. `creditPoints`：活动完成报告审核通过后发放积分。
2. `debitPoints`：志愿者在统一积分商城兑换商品时扣减积分。
3. `refundPoints`：兑换取消时返还积分。
4. `getBalance`：按用户 ID 查询链上积分余额。
5. `getTransaction`：按业务幂等键查询单条链上积分交易。

每笔积分交易使用业务幂等键防止重复记账，例如 `completion:{completionId}:earn:{userId}`、`redemption:{redemptionId}:spend:{userId}`、`redemption:{redemptionId}:refund:{userId}`、`migration:init:{userId}`。

### 12.3 强一致业务规则

积分发放、积分消费、积分退款必须先完成链上交易，再更新数据库缓存和业务索引。若 WeBASE 未配置、链节点异常、链上余额不足或合约调用失败，则本次业务操作直接失败，数据库事务回滚。

### 12.4 后台运维能力

超级管理员新增两类后台能力：

1. `POST /api/admin/points/migrate`：将历史 `users.total_points > 0` 的积分余额初始化迁移到链上，接口按 `migration:init:{userId}` 幂等。
2. `POST /api/admin/points/sync-balances`：从链上读取用户余额并修正 `users.total_points` 缓存。

### 12.5 对前端和接口的影响

志愿者积分流水接口 `GET /api/points` 增加 `bizKey`、`digest`、`txHash`、`blockNumber`、`contractName`、`onchainStatus`、`onchainAt`、`chainBalanceAfter` 等字段。新增 `GET /api/points/balance` 用于对比链上余额与数据库缓存余额是否一致。
