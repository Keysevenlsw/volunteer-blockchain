# 接口联调用例

本文档用于本地后端联调，接口基准地址默认使用 `http://localhost:8080`。所有非公共接口都需要请求头：

```http
Authorization: Bearer {{token}}
Content-Type: application/json
```

## 一、联调前置条件

1. 初始化数据库：新库建议执行 [schema.sql](/d:/codes/volunteer-blockchain/src/main/resources/schema.sql)。
2. 启动前配置超级管理员账号，否则无法给普通账号分配 `activity_reviewer` 和 `product_reviewer` 角色。

```properties
rbac.system-admin.username=平台超级管理员
rbac.system-admin.email=admin@example.com
rbac.system-admin.password=Admin123456
```

3. 启动后访问 `POST /api/auth/login` 登录超级管理员，并在 Apifox 环境变量中保存 `systemToken`。
4. 本项目的注册接口只允许创建 `volunteer` 和 `organization_admin`，活动审核员、商品审核员需要先注册为志愿者，再由超级管理员分配角色。

## 二、Apifox 环境变量

建议创建名为 `本地联调` 的环境，配置以下变量：

| 变量名 | 示例值 | 说明 |
| --- | --- | --- |
| `baseUrl` | `http://localhost:8080` | 后端服务地址 |
| `systemToken` | 登录后写入 | 超级管理员 token |
| `orgToken` | 登录后写入 | 组织管理员 token |
| `volunteerToken` | 登录后写入 | 志愿者 token |
| `activityReviewerToken` | 登录后写入 | 活动审核员 token |
| `productReviewerToken` | 登录后写入 | 商品审核员 token |
| `activityId` | 创建活动后写入 | 活动 ID |
| `completionId` | 提交完成报告后写入 | 完成报告 ID |
| `productId` | 创建商品后写入 | 商品 ID |
| `redemptionId` | 兑换商品后写入 | 兑换记录 ID |
| `evidenceBizType` | `completion` | 存证业务类型 |
| `evidenceBizId` | 完成报告 ID | 存证业务 ID |

登录接口返回 `data.token`，可在 Apifox 后置脚本中写入变量：

```javascript
const body = pm.response.json();
if (body && body.success && body.data && body.data.token) {
  pm.environment.set("volunteerToken", body.data.token);
}
```

根据当前登录账号类型，把变量名改为 `systemToken`、`orgToken`、`activityReviewerToken` 或 `productReviewerToken`。

## 三、账号与角色准备链路

### 1. 登录超级管理员

```http
POST {{baseUrl}}/api/auth/login
```

```json
{
  "email": "admin@example.com",
  "password": "Admin123456"
}
```

保存：

```javascript
pm.environment.set("systemToken", pm.response.json().data.token);
```

### 2. 注册组织管理员

```http
POST {{baseUrl}}/api/auth/register
```

```json
{
  "username": "红十字组织管理员",
  "email": "org-admin@example.com",
  "password": "OrgAdmin123",
  "role": "organization_admin",
  "organizationName": "红十字志愿服务队",
  "organizationDescription": "负责社区志愿服务活动"
}
```

注册成功后登录：

```http
POST {{baseUrl}}/api/auth/login
```

```json
{
  "email": "org-admin@example.com",
  "password": "OrgAdmin123"
}
```

保存 `orgToken`。

### 3. 注册志愿者

```http
POST {{baseUrl}}/api/auth/register
```

```json
{
  "username": "志愿者张三",
  "email": "volunteer@example.com",
  "password": "Volunteer123",
  "role": "volunteer"
}
```

登录后保存 `volunteerToken`。

### 4. 注册审核员账号

先注册两个普通志愿者账号：

```json
{
  "username": "活动审核员",
  "email": "activity-reviewer@example.com",
  "password": "Reviewer123",
  "role": "volunteer"
}
```

```json
{
  "username": "商品审核员",
  "email": "product-reviewer@example.com",
  "password": "Reviewer123",
  "role": "volunteer"
}
```

使用超级管理员查询用户：

```http
GET {{baseUrl}}/api/admin/users
Authorization: Bearer {{systemToken}}
```

记录两个审核员账号的 `userId`，然后分配角色：

```http
PUT {{baseUrl}}/api/admin/users/{{activityReviewerUserId}}/roles
Authorization: Bearer {{systemToken}}
```

```json
{
  "roleCodes": ["activity_reviewer"]
}
```

```http
PUT {{baseUrl}}/api/admin/users/{{productReviewerUserId}}/roles
Authorization: Bearer {{systemToken}}
```

```json
{
  "roleCodes": ["product_reviewer"]
}
```

重新登录两个审核员账号，分别保存 `activityReviewerToken` 和 `productReviewerToken`。

## 四、活动发布审核与跨组织参与链路

### 1. 组织管理员创建活动草稿

```http
POST {{baseUrl}}/api/activities
Authorization: Bearer {{orgToken}}
```

```json
{
  "activityName": "社区环保清洁活动",
  "description": "清理社区公共区域垃圾并进行环保宣传。",
  "startDate": "2026-05-10 09:00:00",
  "endDate": "2026-05-10 12:00:00",
  "location": "广州市天河区社区广场",
  "contactName": "李老师",
  "contactPhone": "13800000000",
  "categoryTags": "GENERAL,环保",
  "imagePath": "/uploads/projects/clean.png",
  "maxParticipants": 30,
  "requestedRewardPoints": 30,
  "enrollDeadline": "2026-05-09 18:00:00"
}
```

保存返回的 `data.activityId` 到 `activityId`。此时 `reviewStatus` 为 `draft`，志愿者活动大厅不可见。

### 2. 组织管理员提交活动审核

```http
POST {{baseUrl}}/api/activities/{{activityId}}/submit
Authorization: Bearer {{orgToken}}
```

提交后 `reviewStatus` 为 `pending_review`。

### 3. 活动审核员查看待审核活动

```http
GET {{baseUrl}}/api/admin/activity-reviews?status=pending_review
Authorization: Bearer {{activityReviewerToken}}
```

返回字段中重点看：

```json
{
  "requestedRewardPoints": 30,
  "recommendedPoints": 30,
  "maxPoints": 50,
  "escalationThreshold": 80,
  "escalationRequired": false
}
```

### 4. 普通活动审核通过

```http
POST {{baseUrl}}/api/admin/activity-reviews/{{activityId}}
Authorization: Bearer {{activityReviewerToken}}
```

```json
{
  "status": "approved",
  "approvedRewardPoints": 30,
  "reviewNote": "活动信息真实，积分合理。"
}
```

审核通过后活动进入前台大厅，积分发放以 `approvedRewardPoints` 为准。

### 5. 高积分活动升级复审

如果 `escalationRequired=true`，活动审核员不能直接通过，应先发起升级：

```http
POST {{baseUrl}}/api/admin/activity-reviews/{{activityId}}
Authorization: Bearer {{activityReviewerToken}}
```

```json
{
  "status": "escalated",
  "reviewNote": "申请积分超过复审阈值，提交超级管理员复审。"
}
```

超级管理员复审通过：

```http
POST {{baseUrl}}/api/admin/activity-reviews/{{activityId}}
Authorization: Bearer {{systemToken}}
```

```json
{
  "status": "approved",
  "approvedRewardPoints": 80,
  "reviewNote": "已核验活动规模，批准 80 积分。"
}
```

### 6. 志愿者浏览全平台已审核活动

```http
GET {{baseUrl}}/api/activities/volunteer
Authorization: Bearer {{volunteerToken}}
```

验证点：志愿者不需要加入活动所属组织，也能看到所有 `reviewStatus=approved` 的活动。

### 7. 志愿者报名活动

```http
POST {{baseUrl}}/api/participations/{{activityId}}
Authorization: Bearer {{volunteerToken}}
```

验证点：接口不再校验志愿者默认归属组织是否等于活动所属组织。

### 8. 志愿者提交完成报告

```http
POST {{baseUrl}}/api/completions
Authorization: Bearer {{volunteerToken}}
```

```json
{
  "activityId": {{activityId}},
  "serviceLocation": "广州市天河区社区广场",
  "serviceStartTime": "2026-05-10 09:00:00",
  "serviceEndTime": "2026-05-10 12:00:00",
  "reportText": "完成社区道路和绿化带清洁。",
  "contributionDetails": "清理垃圾 8 袋，协助完成环保宣传。",
  "attachmentPaths": [
    "/uploads/projects/report-1.png"
  ]
}
```

保存返回的 `data.completionId` 到 `completionId`。

### 9. 组织管理员审核完成报告并发放积分

```http
GET {{baseUrl}}/api/reviews?status=pending
Authorization: Bearer {{orgToken}}
```

```http
POST {{baseUrl}}/api/reviews/{{completionId}}
Authorization: Bearer {{orgToken}}
```

```json
{
  "status": "approved",
  "reviewNote": "服务记录真实，审核通过。"
}
```

验证点：

- 志愿者获得的积分等于活动 `approvedRewardPoints`。
- 组织管理员审核通过后会生成 `blockchain_evidences` 存证索引。
- 后台异步调用 WeBASE 上链；未配置链环境时可能出现上链失败状态，可后续重试。

### 10. 志愿者查看积分流水

```http
GET {{baseUrl}}/api/points
Authorization: Bearer {{volunteerToken}}
```

预期存在一条：

```json
{
  "transactionType": "earned",
  "referenceType": "completion",
  "referenceId": {{completionId}}
}
```

## 五、统一商城链路

### 1. 组织管理员创建商品草稿

```http
POST {{baseUrl}}/api/products
Authorization: Bearer {{orgToken}}
```

```json
{
  "productName": "志愿服务纪念水杯",
  "productDescription": "统一积分商城兑换商品，由所属组织履约。",
  "price": 20,
  "stock": 100,
  "imagePath": "/uploads/products/cup.png"
}
```

保存 `data.productId` 到 `productId`。

### 2. 组织管理员提交商品审核

```http
POST {{baseUrl}}/api/products/{{productId}}/submit
Authorization: Bearer {{orgToken}}
```

### 3. 商品审核员审核通过

```http
GET {{baseUrl}}/api/admin/product-reviews?status=pending_review
Authorization: Bearer {{productReviewerToken}}
```

```http
POST {{baseUrl}}/api/admin/product-reviews/{{productId}}
Authorization: Bearer {{productReviewerToken}}
```

```json
{
  "status": "approved",
  "reviewNote": "商品信息完整，可以进入统一商城。"
}
```

### 4. 志愿者查看统一商城

```http
GET {{baseUrl}}/api/products/volunteer
Authorization: Bearer {{volunteerToken}}
```

验证点：只返回全平台 `reviewStatus=approved` 的商品，不按组织拆分商城。

### 5. 志愿者兑换商品

```http
POST {{baseUrl}}/api/redemptions
Authorization: Bearer {{volunteerToken}}
```

```json
{
  "productId": {{productId}}
}
```

保存 `data.id` 到 `redemptionId`。

### 6. 商品所属组织履约

```http
GET {{baseUrl}}/api/redemptions/organization
Authorization: Bearer {{orgToken}}
```

发放商品：

```http
POST {{baseUrl}}/api/redemptions/{{redemptionId}}/status
Authorization: Bearer {{orgToken}}
```

```json
{
  "status": "fulfilled"
}
```

取消兑换并回滚积分、库存：

```json
{
  "status": "cancelled"
}
```

## 六、公开存证查询链路

### 1. 公开查询存证列表

```http
GET {{baseUrl}}/api/public/evidences?limit=20&status=success&keyword=社区
```

查询参数：

| 参数 | 是否必填 | 说明 |
| --- | --- | --- |
| `limit` | 否 | 最大返回条数，默认 `20` |
| `status` | 否 | 上链状态，如 `pending`、`success`、`failed` |
| `organizationId` | 否 | 按组织筛选 |
| `keyword` | 否 | 按活动、组织、志愿者关键词筛选 |

### 2. 公开查询单条存证详情

```http
GET {{baseUrl}}/api/public/evidences/completion/{{completionId}}
```

预期返回摘要、交易哈希、区块高度、合约名称、审核人和校验结果。

### 3. 登录用户查询单条存证

```http
GET {{baseUrl}}/api/evidences/completion/{{completionId}}
Authorization: Bearer {{volunteerToken}}
```

### 4. 组织管理员或超级管理员重试上链

```http
POST {{baseUrl}}/api/evidences/{{evidenceId}}/retry
Authorization: Bearer {{orgToken}}
```

## 八、关键异常用例

| 用例 | 操作 | 预期 |
| --- | --- | --- |
| 未审核活动不可见 | 活动创建后不提交或不审核，志愿者查询 `/api/activities/volunteer` | 不返回该活动 |
| 非同组织可报名 | 志愿者没有默认归属组织或归属其他组织，报名已审核活动 | 报名成功 |
| 高积分必须复审 | `requestedRewardPoints` 超过 `escalationThreshold`，活动审核员直接 `approved` | 返回失败 |
| 发放积分取批准值 | 申请 100、批准 30，完成报告通过 | 志愿者只增加 30 分 |
| 审核人不能审核自己组织 | 审核员账号绑定了提交组织，审核本组织活动或商品 | 返回失败 |
| 商品未审核不可兑换 | 商品为 `draft` 或 `pending_review`，志愿者兑换 | 返回失败 |
| 取消兑换回滚 | 组织将兑换记录改为 `cancelled` | 用户积分返还，库存加回 |

## 九、链上积分系统联调用例

### 1. 配置积分合约

部署 `contracts/VolunteerPointsLedger.sol` 后，在后端配置中填写：

```properties
webase.points-contract-name=VolunteerPointsLedger
webase.points-contract-address=0x你的积分合约地址
webase.points-contract-abi=classpath:blockchain/VolunteerPointsLedger.abi.json
```

如果 `webase.front-url`、`webase.user-address` 或 `webase.points-contract-address` 为空，发放积分、兑换扣分、取消兑换退款都会直接失败，数据库事务不会提交。

### 2. 历史积分迁移

```http
POST {{baseUrl}}/api/admin/points/migrate
Authorization: Bearer {{systemToken}}
```

预期返回：

```json
{
  "success": true,
  "data": {
    "scannedUsers": 3,
    "migratedUsers": 2,
    "skippedUsers": 1,
    "failedUsers": 0
  }
}
```

接口按 `migration:init:{userId}` 幂等，重复执行不会重复给同一用户初始化加分。

### 3. 查询链上余额与缓存一致性

```http
GET {{baseUrl}}/api/points/balance
Authorization: Bearer {{volunteerToken}}
```

预期返回：

```json
{
  "success": true,
  "data": {
    "userId": 2,
    "chainBalance": 30,
    "cachedBalance": 30,
    "consistent": true,
    "checkedAt": "2026-05-10 12:00:00"
  }
}
```

### 4. 同步链上余额缓存

```http
POST {{baseUrl}}/api/admin/points/sync-balances
Authorization: Bearer {{systemToken}}
```

该接口会读取链上 `getBalance(userId)`，并修正 `users.total_points` 缓存。

### 5. 检查积分流水链上字段

```http
GET {{baseUrl}}/api/points
Authorization: Bearer {{volunteerToken}}
```

积分流水应包含：

```json
{
  "transactionType": "earned",
  "bizKey": "completion:1:earn:2",
  "digest": "sha256摘要",
  "txHash": "0x...",
  "blockNumber": "123",
  "contractName": "VolunteerPointsLedger",
  "onchainStatus": "success",
  "chainBalanceAfter": 30
}
```

### 6. 链上积分异常用例

| 用例 | 操作 | 预期 |
| --- | --- | --- |
| 积分合约未配置 | 审核完成报告通过 | 接口失败，完成报告状态和用户积分不变 |
| 链上余额不足 | 志愿者兑换商品 | 接口失败，不扣库存，不创建成功兑换流水 |
| 重复业务键 | 重复执行同一笔链上积分交易 | 合约拒绝重复记账 |
| 缓存不一致 | 手动修改 `users.total_points` 后查余额 | `GET /api/points/balance` 返回 `consistent=false` |
